# Pi（>=0.80.6 <0.82.0）+ 飞书测试应用真实冒烟方案

当前仓库环境没有真实 Pi/飞书租户，因此以下步骤必须由测试者在真实环境执行并保存截图、Pi 日志和飞书消息链接作为最终证据。

## 准备

1. 安装 Node.js 20+ 和兼容版 Pi（`>=0.80.6 <0.82.0`，推荐 0.81.1），确认 `pi --version` 在范围内。
2. 从本仓库执行 `npm ci && npm run typecheck && npm test && npm pack --dry-run`。
3. 使用生成的 tarball 安装：`pi install ./pi-feishu-bridge-2.0.18.tgz`（版本以 `package.json` 为准），启动 Pi，确认扩展无加载错误。
4. 创建飞书测试应用，按 `docs/permissions.md` 开通权限、事件和 CardKit，发布测试版本。
5. 使用 allowlist 配置测试用户和 chat；准备第二个未授权账号。先运行 `/feishu doctor`，确认凭据、连接和安全警告符合预期。

## 逐项测试

每项记录：输入、预期、实际、卡片/message ID、最终状态、是否内容丢失、是否完整重复、Reaction 是否移除。

1. 短回答：发送“用两句话介绍当前项目”。应只创建一张卡，delta 顺序与最终文本一致，封卡并更新摘要。
2. Thinking：使用会产生 thinking 的模型。默认只显示轮次；开启 `showThinking` 后才显示正文。
3. 单工具：要求读取一个文件。工具开始/结束状态正确，Typing Reaction 最终移除。
4. 多轮多工具：执行至少三个工具，含两个同名并行调用。时间线顺序真实且状态不串。
5. 工具失败：执行一个确定失败的命令。卡片显示工具失败，但 Agent 可继续并最终封卡。
6. LLM API 错误：临时使用无效 provider key 或测试代理注入 5xx。卡片进入失败终态且不重复完整回答。
7. 主动停止：生成长回答期间发送 `/stop`。卡片显示已停止，队列清空，Reaction 移除。
8. Rollover：临时把 `maxAnswerElementChars` 调到 1000，生成超过 3000 字且包含 fenced code 的回答。应创建续卡，代码围栏可读，拼接内容无丢失。
9. 封卡与摘要：确认最后卡 `streaming_mode=false`，聊天列表摘要不再显示“处理中”。
10. 访问控制：未授权账号发送文本、命令和点击澄清卡。不得进入 Pi；响应不得泄露模型、App ID、上下文或状态。
11. 媒体双向：分别发送图片、文件、音频、视频给 Bot；再让 Pi 调用文本/图片/文件主动发送工具。检查下载和上传均成功。
12. 交互澄清：触发 `ask_feishu`，授权用户选择成功；再验证未授权点击、重复点击、超时和 `/stop` abort 不会卡住工具。

## 故障注入补充

若测试代理可重写飞书响应，分别注入 300313、300309、300305、429、500、网络超时、230011/231003。确认有限重试、元素裁剪、尾部 fallback、撤回停止后续调用，并核对 `/feishu monitor` 计数。

## 通过标准

- 回答无内容丢失、无完整重复。
- 每张卡最终处于 completed/failed/aborted/terminated 之一。
- 失败有可见反馈，未授权用户不能控制 Pi。
- WebSocket、媒体收发、Reaction 和三个主动发送工具相较 1.x 不退化。
- 将全部证据填入 `docs/acceptance-checklist.md` 后才可发布。
